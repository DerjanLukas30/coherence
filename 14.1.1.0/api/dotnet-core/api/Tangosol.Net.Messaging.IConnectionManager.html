Interface IConnectionManager

The IConnectionManager is the base SPI (Service Provider Interface) for both IConnectionAcceptor and IConnectionInitiator implementations.

Remarks

Implementations of this interface use a provider-specific mechanism to establish a bi-directional communication channel between two endpoints, represented by an IConnection. Some implementations restrict data transfer between endpoints within a single JVM, whereas others enable two processes to exchange data. Advanced implementations allow communication between processes on different machines, for example using TCP sockets or JMS.

Before an IConnection can be established between a IConnectionInitiator (client) and IConnectionAcceptor (server), one or more IProtocol instances must be registered with the IConnectionManager on each. During IConnection establishment, the IConnectionInitiator sends information about each registered IProtocol. A compatable set of IProtocol instances (or superset) must be registered with the acceptor's IConnectionManager in order for the IConnection to be established.

Establishing an IConnection is assumed to be a heavyweight operation that may allocate significant resources within and outside the JVM. For example, a TCP-based implementation of this interface may implement an IConnection using a persistent Socket connection with a remote server. However, once established, successive uses of the same IConnection should be relatively lightweight. In other words, an IConnection object, once opened, should appear to be persistent from the perspective of the user until closed. Additionally, underlying transports used by implementations must be both reliable and ordered.

Once an IConnection is established, either client or server may open an IChannel to an IReceiver registered by its peer and use it to send and receive IMessage objects to/from the peer.
